# Creative Ops Operational Intelligence

## AI Specialist Evaluation Build — Samantha Revita

This public evaluation package is not positioned as a generic prompt library, retrieval wrapper, dashboard, or policy checklist.

It presents an operational intelligence architecture concerned with a harder question:

```text
When may an AI-assisted movement become operationally real?
```

That question requires more than generation quality. It requires visible standing.

```text
What material basis is active?
Which prompt lineage and provider route participated?
What conditions govern the movement?
What witness records the current standing?
What change requires revalidation?
How can the path be replayed later?
```

## Core proposition

```text
Generation is not the same as operational standing.

An AI-assisted artifact becomes usable only when its material basis,
lineage, conditions, review state, and revalidation path are visible.
```

The familiar technical components—documents, retrieval, prompts, provider routes, automation, model adaptation, and human review—are participating dimensions of the larger system. They are not the architecture itself.

## Why this is relevant to AI Specialist work

The architecture expresses practical AI Specialist capability at the operating-model level:

- prototype AI workflows for creative, technical, and operational teams;
- structure RAG-ready material bases with provenance, authority, versioning, and use scope;
- maintain prompt lineage, evaluation conditions, and provider-route transparency;
- build lightweight interfaces and automation around visible state;
- evaluate commercial and open-source tools by task fit, data boundaries, failure behavior, cost, observability, and release conditions;
- establish model adaptation and LoRA evaluation paths when adaptation is justified;
- enable teams with clear practices for review, escalation, and revalidation;
- turn experimental capability into systems that can be understood, reviewed, and safely used.

## Boundary

This repository remains a public demonstration. It does not include private system internals, credentials, customer material, live provider calls, or a claim of a hosted production service.

The visible work demonstrates architecture, interaction design, implementation judgment, and operational consequence reasoning.
