# Resolve what may become real.

> **Generation is not operational standing.**
>
> An AI-assisted movement may become real only when the active material basis, lineage, governing conditions, witness, review state, and revalidation path are visible.

```text
PROPOSED MOVEMENT
        │
        ▼
ACTIVE MATERIAL FIELD ─── LINEAGE ─── PROVIDER ROUTE
        │                     │               │
        └────────── OPERATIONAL RESOLUTION ───┘
                              │
                              ▼
                    WITNESS + CONDITIONS
                              │
                              ▼
                     REVIEWED STANDING
                              │
                              ▼
                  DRIFT → REVALIDATE → REPLAY
```

---

## The governing question

```text
When may an AI-assisted movement bind consequence?
```

This repository is not a prompt showcase, a retrieval wrapper, a chatbot demo, a policy checklist, or a dashboard abstraction.

It is a public evaluation build for an **operational intelligence architecture**: a visible system for determining what may influence a proposed movement, whether the movement has enough standing to proceed, what evidence records that standing, and what later change makes it no longer current.

## What becomes visible

| Dimension | Operational question |
|---|---|
| **Material standing** | What sources, authorities, constraints, and versions are active? |
| **Lineage** | Which prompt, route, and declared system path participated? |
| **Resolution** | Is the active field sufficient for this proposed movement? |
| **Witness** | What records the basis, reasons, and conditions of the result? |
| **Reviewed standing** | Is this merely generated—or permitted to stand operationally? |
| **Continuity** | What later change makes the prior standing stale? |
| **Replay** | How can the path be reconstructed without rewriting history? |

## Executable proof

The public examples run with the Python standard library only.

```bash
python3 examples/negative_cases.py
python3 examples/operational_lifecycle.py
python3 examples/provider_route_adapter.py
python3 evaluate.py --task tasks/creative.json --record samples/runs/run_001_human.json
python3 -m unittest tests/test_basic_grader.py
```

| Example | Demonstrates |
|---|---|
| [`examples/negative_cases.py`](./examples/negative_cases.py) | Failure-closed behavior: missing basis, inactive material, unauthorized use → `HELD` |
| [`examples/operational_lifecycle.py`](./examples/operational_lifecycle.py) | Active field → `MAY_BIND` → reviewed standing → material drift → `REVALIDATION_REQUIRED` |
| [`examples/provider_route_adapter.py`](./examples/provider_route_adapter.py) | Provider-route boundary, data classification, source-basis and prompt-lineage conditions, dry-run observability |
| [`EVALUATION_SYSTEM.md`](./EVALUATION_SYSTEM.md) | Tasks, deterministic scores, refusal conditions, evaluation schema, run artifacts, and test path |

The code does not simulate a magical model. It demonstrates the operational state that must surround one.

## Capability field

This build demonstrates the ability to:

```text
prototype AI-supported operational systems
structure RAG-ready material authority and provenance
maintain prompt and provider-route lineage
build lightweight Python and browser-based interfaces
separate generation from reviewed standing
create deterministic hold, escalation, drift, and revalidation paths
create task catalogs, schemas, graders, replayable run artifacts, and test surfaces
model safe provider-adapter boundaries
reason about LoRA and model adaptation as evidence-bearing change
translate architecture across creative, support, technical, and organizational contexts
```

The components are not the identity of the system.

```text
RAG is a mechanism.
Prompting is a mechanism.
Provider integration is a mechanism.
LoRA is a mechanism.

Standing is the system question.
```
